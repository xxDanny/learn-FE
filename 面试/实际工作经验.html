    实际工作经验

    # 实际工作经验

    # H5 如何首屏优化？

      1. 路由懒加载
      • 用于 spa （不适合于 mpa）
      • 路由拆分，优先保证首屏加载
      1. 服务端渲染 ssr
      • 传统前后端分离 SPA 渲染页面的过程很复杂：第一步从服务器获取 html 加载 js，第二部运行 js，得知需要再运用 ajax 向服务器请求一些数。第三部渲染页面。需要三步，然而 ssr 只需要一步骤。
      • ssr 渲染页面过程很简单

    ssr 是第一代 web 技术，比如 PHP，ASP，JSP，并不是什么新技术。后面前端技术不断发展，大家都开始使用前后端分离来开发网站，但是逐渐又出现了性能问题。大家有想起了 ssr，但是这个时候，大家已经采用了 vue，react 等前端框架，再用 PHP，JSP 又不兼容了 ，所以大家又开始造轮子开发出了 nuxt.js 和 next.js

    1. APP 预取

    如果 H5 在 APP Webview 中展示，可以使用 app 预取。用户访问列表时，App 预加载文章首屏内容

    用户进入 H5 页，直接从 App 中获取内容，瞬间展示首屏

    app 里面有一个 webview 这样我们将能够访问 H5 页面。

    1. 分页 针对列表页，默认只展示第一页

    2. 图片懒加载 注意：提前设置好图片尺寸，尽量直重绘不重排

    3. hybrid 提前将 html css js 下载到 app 内部。在 app web view 中使用 file：//协议加载页面文件。 省去了从网络下载 html/js 的时间

    性能优化要配合分析优化统计和评分，做事情要有结果。把网页打开时间从 1s 优化到了 0.5s 性能优化也要配合体验：比如骨架屏，loading 动画等

    # 后端一次性返回 10w 条数据。

    1. 首先这个设计不合理。第一数据加载需要很多时间，页面的渲这些数据也需要很长的时间。我们可以分页返回。
    2. 首先我们思考一下浏览器是否能处理 10w+数据。js 处理是没有问题的，但是 dom 渲染会非常卡顿

    解决方案：自定义中间层

    1. 自定义 nodejs 中间层，获取并拆分这 10w+数据。前端对接 nodejs 中间层，成本高
    2. 前端解决：虚拟列表。可以借用第三方 lib
      • 只渲染可视区域内的 DOM。其他隐藏区域不显示，只用 div 撑起高度，随时创建销毁 dom
      • vue-virual-scroll-list

    # 前端常用设计模式？使用场景？

    在说设计模式之前，我们要先说说设计原则。需求指导设计，设计指导开发

    最重要的原则：开放封闭元素

    对拓展开放。 对修改封闭封闭。

    1. 工厂模式，用工厂函数创建实例，隐藏 new

    比如 jquery $函数

    1. 单例模式

    产生唯一实例：比如 vuex 的 store

    1. 代理模式

    使用者不能直接访问一个对象，而是访问一个代理层

    vue3 响应式使用 proxy 实现响应式

    1. 观察者模式

    一个主题，一个观察者，主题变化之后触发观察者执行

    btn.addEvent('click', ..)

    1. 发布订阅模式

    自定义事件，在 vue 组件通讯非常常见

    event.on('event-key', function() {

})

event.emit('event-key')

    # 观察者模式和发布订阅者模式区别

    观察者模式是 subject 和 observer 直接绑定

    比如按钮绑定事件

    发布订阅者模式 publisher 和 observer 互相不认识，需要中间媒介 event channel

    比如 evnetbus 自定义事件

    # 实际工作中，进行过哪些 vue 优化

    1. v-if 和 v-show：大部分情况使用 v-if 更好，不要过度优化
    2. v-for 使用 :key
    3. 使用 computed 缓存。比如缓存未读消息。应用场景：组件数据比较大，计算比较繁琐，可以用 computed 进行数据的缓存
    4. keep-alive 缓存组件：比如像频繁切换的组件比如 tabs，如果用 v-if 会销毁创建。
    5. 异步组件：针对比较大的组件，如编辑器，复杂表格，复杂表单等。进行拆包，需要时异步加载，不需要时不加载。减少主包体积，首屏加载也会变快
    6. 路由懒加载。项目比较大进行路由拆分，保证首屏先加载
    7. 服务端渲染 使用 nuxt.js. 按需优化，成本比较高

    # 使用 vue 遇到过的坑

    1. 内存泄漏：一些系统泡着跑着就很慢。因为组件全局变量，自定义事件绑定没有解绑

    2. vue 响应式缺陷

      • 新增 data 属性用 Vue.set
      • data 删除属性用 Vue.delete
      • 无法直接修改数据 `arr[index] = valu3.

    3. 路由切换 scroll 到顶部。是 spa 的通病，不仅仅是 vue。

      • 列表页，滚动到第二屏，点击进入详情页，再返回列表页（此时组件重新渲染）就 scroll 到顶部

    解决：缓存列表页数据和 scrollTop 值。切换路由就是组件的销毁和渲染。

    日常工作中除了开发打包上线，最基本的要求，我们后续还需要考略运维，监控，性能优化等等。不是做完就完事，还要考虑怎么迭代升级重构。

    处理报错 处理统计，处理运维，不能光考虑开发

    # 项目中遇到那些难点？是怎么解决的？

    1. 遇到问题进行积累
    2. 遇到问题记录一下，写文章复盘一下。我会复现场景，解决方案。

    如何将动态图表输出为视频

    描述问题：背景+现象+造成的影响

    1. 上线登录服务器密码之后，通过 amis 文档上线报错。结果的 linux 服务需要重启，通过 linux 进行重启。

    问题：编辑器只能回显 JSON 格式，不支持老版本的 html 格式 解决：使用 html 属性表示 JSON 数据，即可解决这个问题 成长：要考略完整的输入输出+ 考略旧版本的用户 +参考其他产品

    # 项目中有没有发生过沟通的冲突（和其他角色）？如何解决的

    需求变更：PM 或者老板提出了新的需求 时间延期：上游或者自己延期了 技术方案冲突：如感觉服务端给的接口格式不合理

    正视冲突 从个人心理上，不要看到冲突就心烦，要拥抱变化，正视冲突。冲突是项目的一部分，就像 bug 一样，心烦没用。例如，PM 过来说要修改需求，你应该回答：“可以呀，你组织个会议讨论一下吧，拉上各位领导，因为有可能会影响工期。”再例如，自己开发过程中发现可能会有延期，要及早的汇报给领导：“我的工期有风险，因为 xxx 原因，不过我会尽量保证按期完成。”千万不要不好意思，等延期了被领导发现了，这就不好了。

    解决冲突 合作引起的冲突，最终还是要通过沟通来解决。一些不影响需求和工期的冲突，如技术方案问题，尽量私下沟通解决。实在解决不了再来领导开会。需求变更和时间延期一定要开会解决，会议要有各个角色决定权的领导去参与。注意，无论是私下沟通还是开会，涉及到自己工作内容变动的，一定要有结论。最常见的就是发邮件，一定要抄送给各位相关的负责人。这些事情要公开，有记录，不要自己偷偷的就改了。

    如何规避冲突 预估工期留有余地定期汇报个人工作进度，提前识别风险
