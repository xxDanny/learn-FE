    第一个面试官：我看你硕士研究方向是自然语言处理，那你觉得国内哪些公司在这方面比较厉害？国内和国外的文章都有什么特点，哪个对你的启发最大？

    我的缺点可能是性格比较内向，缺乏自信。公开说话会比较紧张，但是现在我会有意识的联系自己语言组织和逻辑表达能力。更积极主动的去跟人沟通，培养自己的自信心。 国内我有看我也有看一些国外的脱口秀，TED 演讲，看看优秀的人是怎么说话，实习的时候，同事姐姐给我推荐《自卑与超越》这本书接受自己的不足，明白自己的自卑的原因。非暴力沟通，奇葩说。

    我的优点，做事情我一旦确定目标，我就会全力以赴。大学期间，我们专业中外合作专业，学的是雅思英语，课程考核经常就会要求和外交一对一进行英语沟通。但是我的口语并不好，但是后来我下定决心要提高自己的口语能力。我就会去有意识的看美剧，听 TED，当时还下载了英语流利说。每天我都会在上面联系半个小时。到后来，我考研复试的时候，得到了老师的赞赏。

    找工作最看重？ 一下三点，第一是否能胜任这份工作，我过往的经历可以在这个岗位发挥更大的优势。第二是工作的成长空间，在工作中接触的项目能学到什么，提升自己哪些专业能力。第三是公司的前景和企业文化，我在面前前有了解贵公司主要是民生银行的研发中心，承担者民生银行数字化转型和企业架构升级的研发工作，有比较好的行业发展前景和企业文化。这是我投递咱们公司的原因。相信我可以就很快融入团队之中

    你认为互联网金融对传统银行业的冲击和机遇有哪些？

    答：互联网金融对传统金融的冲击，有以下三点：1. 对服务模式的冲击，传统的服务模式是人对人，而互联网金融是数据对人。第二是成本的冲击。成本也不一样，有些成本高，有些成本低，但整体来说，互联网金融成本是较低的。第三是业务的冲击，打破了时间和地点的限制。

    机遇也是显而易见的，主要体现在两者可以合作共赢，因为他们有各自的特点，如果他们两者可以结合，一定会比只有单方面更好。第二，我们可以借机降低成本，可以利用互联网的技术 更好的去服务自身。比如以前银行主要使用的是传统方式，但是随着科技发展，以及互联网金融的冲击，很多银行都推出了自己的 app，小程序，推出了自己的网上金融产品。第三可以充分利用大数据，通过大数据的分析就可以对客户进行画像，比如这个人喜欢花钱，流水很大，可以通过大数据向他推出适合他的金融理财产品

    一、如何更好的学习（书中第二、三、四、七章）

    我们的大脑有两种截然不同的 思维模式——专注模式和发散模式。你会在两种模式之间装换，择一而用。）学习时两种思维模式相互切换帮你掌握新知识。先专注理解、记忆，再发散把新学的概念整合到记忆中，最后再专注复习新学的知识。这里有两点需要注意：首先，复习的时间最好不要超过一天，因为超过一天，新学的概念很容易从脑海中消失；其次，复习时最好使用先回想复习，再阅读学习材料查缺补漏。而不是只重复阅读学习材料。

    番茄学习法也就是来自于这本书中。

    1. 工作记忆（组块）、长期记忆以及组块的构建

    如何将工作记忆转为长期记忆？间隔回想重复，根据人的记忆曲线来进行回想重复。推荐软件 AnkiDroid。

    专注模式下，我们的注意力连接在我们的工作记忆上；而发散模式下，我们的注意力则分散在记忆各处，这也是为什么发散模式更有创造力的原因

    帮助构建组块的其他方法：1. 回想学习材料，即提取练习。不要仅仅重复阅读。2. 常练不忘，不能只看学习材料，还要找习题练习。3. 离开平时的学习地点，到其他地方回想材料，这可以让人从不同角度看待问题，增进理解。4. 穿插学习，把不同的知识混在一起学习，这使我们知道该何时使用这个知识。 这篇总结里涵盖了本书所有章节里提到的克服拖延症的方法：

    1. 克服拖延症

    每晚写下第二天的计划。2. 每天对自己承诺要完成一定的惯常事物和任务。3. 让自己身处干扰少的地方学习。4. 相信自己的新时间系统，该学习时学习，该休息时休息，不要有负罪感。5. 做一些后备计划。6. 细化工作，确保自己得到足够的奖励。7. 先从最困难的事做起。8. 意识到刚投入工作有点负面情绪是特别正常的现象

    # 图表动画视频如何生成？

    Canvas 如何快速导出 mp4? (opens new window)

    1. 如果我们用 Canvas 实现了一些动画效果，需要将它回放出来，很多人通常就是用录屏工具将屏幕内容录下来播放，很少有人知道，Canvas 可以直接通过现代浏览器支持的 Media Streams API 来转成视频。

    Canvas 对象支持 captureStream 方法，这个方法会返回一个 MediaStream 对象。然后，我们可以通过这个对象创建一个 MediaRecorder 来录屏。

    我开始认为是通过 Canvas 逐帧绘制，然后用 CaptureStream + MediaRecorder 导出 mp4，但是我尝试后，导出的视频帧率无法控制

    如果一个 canvas 操作完成后面不在变动的话，录制的时候会发现最终的视频并没有想要的长度。这是因为如果 canvas 不动的话后续就不再录制了。\

    解决办法也很简单，找到 canvas 中的一个元素，然后循环改变这个元素的透明度就可以（为了减少对视觉的影响，可以让透明度在 0.99 和 1 之间变化，这个目的就是让 canvas 一直在操作。当然用其他的方案也是可以）

    但是有面临另外一个问题，就是 canvas 绘制过程如果卡了，录制的视频也会是卡的。

    与音频结合 (opens new window)

    Canvas 录制好的视频，我们还可以将它和音频结合，方法是通过 ffmpeg 的 Web 端来合成。

    1. 图片序列 + ffmpeg 性能实在捉急 虽然在导出速度上不占优势，所需时间往往约等于视频时长，但因为不占用服务器资源 最终更换了方案，切换到在服务端用 puppeteer + GPU 服务器来渲染，非常完美。

    2. 使用 WebCodecs (opens new window) 来实现前端的导出并且因为前端提前缓存了编辑的素材，产物也是保存在前端，节省了素材缓存和产物下载的时间，在短视频场景下，同服务端导出相比，前端导出可以更快。

    黑客在网页中潜入一段恶意代码，用户在访问这个网页的时候，恶意代码就会执行。一个最常见的 XSS Payload 就是盗取用户的 Cookie, 从而发起 Cookie 劫持攻击。Cookie 中，一般会保存当前用户的登录凭证，如果 Cookie 被黑客盗取，以为着黑客有可能通过 Cookie 直接登进用户的账户，进行恶意操作

    # 跨域

    # cors

    跨域就是当前页面所在的源和请求目标地址的源不一致。针对一定要跨域的情况，浏览器给除了一个比较正式的解决方案。一个跨域请求可能会触发一个额外的 options 请求。所以把是否会触发 options 请求把跨域请求分为简单请求和非简单请求

    简单请求只需要在服务端的 HTTP 响应头里添加 acces-control-allow-origin 为*

    通常你发出的请求只有请求方法和目标地址，这就是一个简单请求（GET HEAD POST）。但是如果 content-type 为 json 就不是简单请求

    因为一些跨域请求需要在请求头中添加额外的参数，或者传输的数据是 json 或者其他类型。这就是一个复杂请求。浏览器在发起非简单跨域请求的时候，会额外发送一个预检请求，这个请求的请求方法是 options。

    服务端怎么应答预检请求呢？

    服务端通过一下六个字段告诉浏览器本次请求什么操作是允许的

    • Access-control-allow0-origin ：允许的源
      • 当发起的请求是一个需要携带 cookie 的请求时。服务端返回的允许源字段不能为*
    • Access-Allow-Methods 用来控制允许的请求方法（get HEAD POST，DELETE，OPTIONS，CONNEXT，TRACE）
    • ACCESS-Control-Allow-Credentials 控制是否浏览器发送 cookie 。前后端都需要配置 fetch 设置 credetials 为 true。后端需要添加 access-control-allow-credentials 字段为 true
      • 前提：目标源的 cookie 可以在当前源发送出去。比如在 a.com 请求 api.a.com，这里有一部分 cookie 是可以发送给服务端的。如果在 a.com 请求 b.com。就没有发送 cookie

    • Aceess-Control-Allow-Headers。任何你在使用简单请求之前额外的请求头信息都必须包含在这个字段里面
    • Aceess-Control-Max-age: 控制预检请求有效期
    • Aceess-Control-Expose-Headers 允许能够访问到的头信息 response。headers

    # AXIOS 二次封装

    AXIOS 是一个基于 promise 的网络请求库。代码运行在 node 环境中使用的是 HTTP 模块。运行在浏览器中使用的是 XHR 接口 除了可以无缝衔接两个运行环境，它的另一个特点是能够支持拦截请求和响应

    baseURL 通常用来配置通用的前缀。而 url 用来配置后面的路径等

    axios.defaults.baseURL = 'https://www.xxx.com'
    1

    axios.defaults 是对 axios 实例进行的通用配置，也就是会在每个 axios 实例使用到。因此我们通常把域名放在 baseUrl 中，url 保存相对请求路径

    axios 可以依据数据类型，自动推断出 content-type

    params 是一个可以拼接 url 的属性。或者说是用来处理 url query 部分的属性。能把 params 中的键值对序列化成 query 字符串然后拼接到 url 尾部。

    timeout 配置请求超时时间的属性，超时事件触发后会抛出一个 promise reject，需要 catch 捕捉

    withCredentials：处理请求是否携带 cookie。因为 axios 底层使用的是 xhr。这个属性对于同域的请求是无效的。也就是说对于同域请求即使设置为 false，也照样会发送 cookie。对于不同域名的请求，设置为 true 的时候，要注意在后端配置好跨域策略

    • Access-Control-Allow-Origin 必须指定了当前域名，不能够为 *

    在 axios 中会有请求拦截和响应拦截，请求拦截函数会在请求发出之前执行，它的参数是合并之后的配置信息
